\section{Introduction}

This report is designed for two goals at the same time. The first goal is to teach the full learning path of the course in a clear order, from software architecture (\emph{architecture logicielle}) and web services (\emph{services web}) to containers (\emph{conteneurs}), Kubernetes, cloud infrastructure, event streaming, and infrastructure as code. The second goal is to connect that theory to the actual repository in \texttt{projet-programmation-distribuee}, where two services already exist: \texttt{service-a} in Flask and \texttt{service-b} in FastAPI.

The course programme (\emph{programme}) extracted from the local course documents emphasizes DevSecOps (\emph{DevSecOps}), cloud-native engineering, microservices, REST or gRPC communication, cloud infrastructure, Docker, Kubernetes, Terraform, and project-based evaluation. \section{Distributed Programming Big Picture}

Distributed programming (\emph{programmation distribuee}) begins when one program is not enough. Instead of one isolated executable, we now have multiple components that communicate through the network. These components may run on different machines, start at different times, fail independently, scale independently, and be updated independently. That is why architecture matters much more in distributed systems than in small local scripts.

\ClientServerFigure

\subsection{Client, service, and data roles}

The client (\emph{client}) is responsible for asking for work or information. The service (\emph{service}) receives the request, applies business logic, and returns a response. The data layer stores persistent information or streams events. The course material on the big picture of web services and the software architecture lecture both reinforce this same separation of concerns.

\begin{lstlisting}[style=terminal, caption={Mental model for one request-response cycle}]
Client -> HTTP request -> Service -> Database or stream -> HTTP response
\end{lstlisting}

\LearningCallout{mintwash}{
\textbf{Why this matters.}\\
In a distributed system, the network is never free. A function call inside one process is not the same as a remote call across services. The moment you leave one process boundary, you must think about latency, failures, timeouts, contracts, and serialization.
}

\subsection{REST, RPC, and web services}

The course programme explicitly mentions web services in REST and optionally gRPC, plus remote procedure call (\emph{appel de procedure distante}). These are not identical styles.

\begin{center}
\begin{tabularx}{\textwidth}{@{}lYY@{}}
\toprule
\textbf{Style} & \textbf{Main idea} & \textbf{Typical use} \\
\midrule
REST & expose resources through HTTP methods and JSON & public APIs, simple service integration \\
RPC / gRPC & call remote procedures with explicit method contracts & internal service-to-service calls, strong schemas \\
Messaging & send events asynchronously through a broker & loose coupling and event-driven systems \\
\bottomrule
\end{tabularx}
\end{center}

\MemoryBox{REST thinks first about resources. RPC thinks first about methods. Messaging thinks first about events.}

\subsection{Worked active learning}

\LearningCallout{skywash}{
\textbf{Question.} Why is a remote call more fragile than a local function call?\\
\textbf{Answer.} Because a remote call crosses the network, depends on another running service, and can fail because of latency, DNS, authentication, timeouts, or service unavailability.\\[0.6em]
\textbf{Question.} Why do distributed systems need explicit contracts?\\
\textbf{Answer.} Because different services and clients must agree on URLs, methods, schemas, and error formats.
}

\section{Software Architecture And Microservices}

The software architecture lecture (\emph{Software Architecture}) and the microservices lecture (\emph{microservices}) naturally belong together. The architecture lecture teaches decomposition, responsibilities, dependencies, and quality attributes. The microservices lecture narrows that general idea into service-oriented decomposition.

\subsection{Monolith versus microservices}

A monolith keeps many functions inside one application. A microservice architecture splits those functions into smaller services, each with a narrower responsibility. This is powerful, but not free.

\begin{center}
\begin{tabularx}{\textwidth}{@{}lYY@{}}
\toprule
\textbf{Criterion} & \textbf{Monolith} & \textbf{Microservices} \\
\midrule
Deployment & one unit & many units \\
Data ownership & often shared & ideally per service \\
Complexity location & inside the codebase & inside code plus network and operations \\
Independent scaling & limited & stronger \\
Failure isolation & weaker & stronger if well designed \\
\bottomrule
\end{tabularx}
\end{center}

\MistakeBox{Microservices are not automatically better. They are better only when the team and the system actually benefit from separate deployment, separate scaling, or separate ownership.}

\subsection{The current repository as a microservice seed}

The current repository is already a valid teaching example of the first microservice step.

\ProjectArchitectureFigure

\begin{lstlisting}[style=code, language=Python, caption={Service A: simple Flask endpoint}]
from flask import Flask, jsonify

app = Flask(__name__)

@app.route("/")
def index():
    return jsonify({"message": "Hello from Service A", "status": "success"})
\end{lstlisting}

\begin{lstlisting}[style=code, language=Python, caption={Service B: FastAPI service calling Service A}]
@app.get("/")
async def index():
    response = await http_client.get(SERVICE_A_URL)
    response.raise_for_status()
    service_a_data = response.json()

    return {
        "service_b": {"message": "Hello from Service B", "status": "success"},
        "service_a": service_a_data,
    }
\end{lstlisting}

\begin{lstlisting}[style=terminal, caption={Expected JSON output from Service B}]
{
  "service_b": {
    "message": "Hello from Service B",
    "status": "success"
  },
  "service_a": {
    "message": "Hello from Service A",
    "status": "success"
  }
}
\end{lstlisting}

\LearningCallout{rosewash}{
\textbf{Important architectural lesson.}\\
Service B is not just another endpoint. It is already acting as an integration layer: it calls another service, checks the response, and returns a combined output. That is exactly the kind of step that turns a small local app into a distributed system.
}

\subsection{Gateway, service mesh, and project milestones}

The project brief suggests a progressive path: one service first, then a gateway, then a second service, then optionally a service mesh. This is a good pedagogical order because each milestone introduces one new kind of complexity instead of all complexity at once.

\begin{enumerate}
  \item One service in local mode.
  \item Package that service as a Docker image.
  \item Run it in Kubernetes.
  \item Add a gateway (\emph{passerelle}) or ingress.
  \item Add a second service and inter-service communication.
  \item Add a database and security rules.
\end{enumerate}

\section{Web Services, REST, And Cloud Integration}

The big-picture, cloud-integration, and REST practice materials all converge on the same core skill: designing and consuming service interfaces correctly.

REST web services (\emph{services web REST}) typically expose URLs, HTTP methods, and JSON payloads.

\subsection{Error handling and contracts}

Integration is not only about the happy path. One service may be down, may return invalid JSON, or may reject a request because of validation or permissions.

\begin{lstlisting}[style=terminal, caption={Typical failure contract}]
502 Bad Gateway
{
  "detail": "Service A unavailable: timeout"
}
\end{lstlisting}

The current \texttt{service-b} already demonstrates a good practice here: it catches \texttt{httpx.HTTPError} and translates it into a \texttt{502} response.

\section{Persistence And JPA}

The course sources \texttt{JPA en.pdf} and \texttt{TP JPA.pdf} cover persistence (\emph{persistance}) and the Java Persistence API. Even if the current repository is Python-based, JPA remains important because it teaches object-relational mapping (\emph{mapping objet-relationnel}), entities, repositories, and transactional thinking.

\subsection{Why persistence matters in distributed systems}

Without persistence, a service only remembers data while the process is alive. As soon as the service restarts, everything disappears. Real distributed systems therefore need a clear data model, durable storage, and a strategy for consistency across service boundaries.

\subsection{JPA mental model}

\begin{center}
\begin{tabularx}{\textwidth}{@{}lY@{}}
\toprule
\textbf{JPA concept} & \textbf{Meaning} \\
\midrule
Entity (\emph{entite}) & Java class mapped to a database table \\
Repository (\emph{repository}) & data-access interface for queries and persistence \\
Transaction (\emph{transaction}) & controlled unit of database work \\
ORM & object-relational mapping between code objects and tables \\
\bottomrule
\end{tabularx}
\end{center}

\begin{lstlisting}[style=code, language=Java, caption={Simple JPA entity example}]
@Entity
public class Order {
    @Id
    @GeneratedValue
    private Long id;

    private String customer;
    private String status;
}
\end{lstlisting}

\begin{lstlisting}[style=code, language=Java, caption={Repository example}]
public interface OrderRepository extends JpaRepository<Order, Long> {
    List<Order> findByStatus(String status);
}
\end{lstlisting}

\LearningCallout{skywash}{
\textbf{Code explanation.}\\
\texttt{@Entity} marks the class as persistent.\\
\texttt{@Id} defines the primary key.\\
\texttt{@GeneratedValue} asks the database or ORM to create the identifier.\\
\texttt{JpaRepository<Order, Long>} gives CRUD operations without writing SQL manually for every simple case.
}

\subsection{JPA practice with answer}

\LearningCallout{mintwash}{
\textbf{Exercise.} Add an email field to a JPA \texttt{Customer} entity and make it unique.\\
\textbf{Solution idea.} Declare the field in the entity and add a uniqueness constraint so the schema protects the data model automatically.
}

\begin{lstlisting}[style=code, language=Java, caption={JPA exercise solution}]
@Entity
public class Customer {
    @Id
    @GeneratedValue
    private Long id;

    @Column(unique = true)
    private String email;
}
\end{lstlisting}

\textbf{Result.} The database schema enforces uniqueness, so the same email cannot be inserted twice.

\section{Docker And Docker Images}

The course sources \texttt{9 - Docker.pdf} and \texttt{Getting started with Docker Images.pdf} are central because containers are the bridge from local code to reproducible deployment.

\DockerFigure

\subsection{Image versus container}

An image (\emph{image}) is a built artifact. A container (\emph{conteneur}) is a running instance of that artifact. This distinction is one of the most important memory points in the whole course.

\MemoryBox{Image = packaged recipe. Container = running instance.}

\subsection{Walkthrough of the current Dockerfiles}

\begin{lstlisting}[style=code, caption={Service A Dockerfile}]
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY app.py .

EXPOSE 5000

CMD ["python", "app.py"]
\end{lstlisting}

\begin{lstlisting}[style=code, caption={Service B Dockerfile}]
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY main.py .

EXPOSE 5001

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "5001"]
\end{lstlisting}

\LearningCallout{mintwash}{
\textbf{Line-by-line explanation.}\\
\texttt{FROM} selects the base image.\\
\texttt{WORKDIR} chooses the working directory inside the container.\\
\texttt{COPY requirements.txt .} copies only dependencies first, which helps reuse image layers.\\
\texttt{RUN pip install ...} installs Python packages.\\
\texttt{COPY app.py .} or \texttt{COPY main.py .} adds the service code.\\
\texttt{EXPOSE} documents the port.\\
\texttt{CMD} gives the default startup command.
}

\subsection{Build and run practice}

\begin{lstlisting}[style=terminal, caption={Service A local container workflow}]
cd service-a
docker build -t service-a .
docker run -p 5000:5000 service-a
\end{lstlisting}

\begin{lstlisting}[style=terminal, caption={Expected response from Service A}]
curl http://localhost:5000/
{"message":"Hello from Service A","status":"success"}
\end{lstlisting}

\subsection{Active learning with solution}

\LearningCallout{rosewash}{
\textbf{Exercise.} Why is it better to copy \texttt{requirements.txt} before copying the whole application?\\
\textbf{Answer.} Because Docker can reuse the dependency-install layer if the code changes but the dependency file does not. This makes rebuilds faster.
}

\section{Kubernetes And Container Orchestration}

The Kubernetes lecture (\texttt{12 - Kubernetes.pdf}) is where one container becomes a managed system. Kubernetes solves scheduling, desired state, replica management, service exposure, rolling updates, and recovery.

\KubernetesFigure

\subsection{Core objects}

\begin{center}
\begin{tabularx}{\textwidth}{@{}lY@{}}
\toprule
\textbf{Object} & \textbf{Role} \\
\midrule
Pod & smallest deployable unit, usually one main container \\
Deployment & keeps the desired number of pod replicas alive \\
Service & stable network endpoint in front of pods \\
ConfigMap / Secret & external configuration and sensitive data management \\
Ingress or Gateway & external routing into the cluster \\
\bottomrule

\begin{lstlisting}[style=code, caption={Simple Kubernetes deployment and service}]
apiVersion: apps/v1
kind: Deployment
metadata:
  name: service-a
spec:
  replicas: 2
  selector:
    matchLabels:
      app: service-a
  template:
    metadata:
      labels:
        app: service-a
    spec:
      containers:
        - name: service-a
          image: your-dockerhub-user/service-a:latest
          ports:
            - containerPort: 5000
---
apiVersion: v1
kind: Service
metadata:
  name: service-a
spec:
  selector:
    app: service-a
  ports:
    - port: 80
      targetPort: 5000
\end{lstlisting}

\begin{lstlisting}[style=terminal, caption={Expected kubectl output}]

service-a-def456   Running

$ kubectl get service

\end{lstlisting}

\MistakeBox{A deployment is not the same thing as a service. The deployment manages pods. The service gives a stable network identity.}

\section{Kafka And Event-Driven Communication}

The Kafka course file introduces asynchronous communication. This matters because not all distributed interactions should be request-response.

\KafkaFigure

\subsection{Why use Kafka}

Kafka is useful when one service should publish an event and several consumers may react later. It supports decoupling, buffering, replay, and scalable consumer groups.

\begin{lstlisting}[style=code, language=Python, caption={Conceptual producer example}]
producer.send("orders.created", value={
    "order_id": 17,
    "customer": "Alice",
    "status": "created",
})
\end{lstlisting}

\begin{lstlisting}[style=code, language=Python, caption={Conceptual consumer example}]
for message in consumer:
    event = message.value
    print("processing", event["order_id"])
\end{lstlisting}

\begin{lstlisting}[style=terminal, caption={Visible output}]
processing 17
\end{lstlisting}

\LearningCallout{mintwash}{
\textbf{When to choose Kafka.}\\
Choose Kafka when services should react to events independently and at different speeds. If every call needs an immediate answer, REST or gRPC is often simpler.
}

\section{Worked Active Learning And Practice}

This section is intentionally practical. Every exercise has a complete answer, an explanation, and an expected visible result.

\subsection*{Exercise 1: explain the role of Service B}
\textbf{Question.} What makes \texttt{service-b} more than just another endpoint?

\begin{lstlisting}[style=code, language=Python]
response = await http_client.get(SERVICE_A_URL)
response.raise_for_status()
service_a_data = response.json()
\end{lstlisting}

\textbf{Answer.} \texttt{service-b} is an integration service. It consumes another service over HTTP and composes the final response.

\textbf{Expected result.}
\begin{lstlisting}[style=terminal]
Service B returns its own message plus Service A data.
\end{lstlisting}

\subsection*{Exercise 2: build a container}
\textbf{Question.} Which Docker command turns a Dockerfile into an image?

\begin{lstlisting}[style=terminal]
docker build -t service-a ./service-a
\end{lstlisting}

\textbf{Answer.} \texttt{docker build} reads the Dockerfile and creates an image tagged \texttt{service-a}.

\subsection*{Exercise 3: read a Kubernetes deployment}
\textbf{Question.} What does the \texttt{replicas} field mean?

\begin{lstlisting}[style=code]
spec:
  replicas: 2
\end{lstlisting}

\textbf{Answer.} Kubernetes should keep two pod instances alive.

\subsection*{Exercise 4: detect a REST contract}
\textbf{Question.} Why is JSON a good data format for REST services?

\textbf{Answer.} Because it is text-based, easy to read, easy to parse in browsers and backend languages, and naturally matches objects, arrays, and key-value structures.

\subsection*{Exercise 5: design a Kafka event}
\textbf{Question.} Write a small event payload for an order creation.

\begin{lstlisting}[style=code]
{
  "event": "order.created",
  "order_id": 17,
  "customer": "Alice"
}
\end{lstlisting}

\textbf{Explanation.} The event name identifies the kind of change, and the payload carries the minimum information that consumers need.

\subsection*{Exercise 6: reason about Terraform}
\textbf{Question.} Why is \texttt{terraform plan} important before \texttt{terraform apply}?

\textbf{Answer.} Because it previews the changes and gives you a chance to catch destructive or incorrect infrastructure modifications before they happen.

\subsection*{Exercise 7: persistence design}
\textbf{Question.} What is one reason a microservice should usually own its own database schema?

\textbf{Answer.} To reduce tight coupling and allow the service to evolve independently.

\subsection*{Exercise 8: graceful failure}
\textbf{Question.} What should \texttt{service-b} do if \texttt{service-a} is unavailable?

\textbf{Answer.} Return a clear error, for example a \texttt{502} response, instead of hanging silently or crashing without explanation.

\subsection*{Exercise 9: cloud reasoning}
\textbf{Question.} Give one reason to deploy the project in a cloud environment.

\textbf{Answer.} Cloud infrastructure makes it easier to provision compute, networking, storage, and managed services without owning physical servers.

\subsection*{Exercise 10: project strategy}
\textbf{Question.} Why does the project brief recommend starting with only one service?

\textbf{Answer.} Because a single working service establishes the first stable baseline. Without that baseline, adding containers, Kubernetes, gateways, and databases makes debugging much harder.
